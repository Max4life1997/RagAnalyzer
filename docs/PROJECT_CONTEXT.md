# RagAnalyzer: контекст проекта

## 1. Какая проблема есть

В проекте нет README или другого входного документа, который быстро объясняет, что делает приложение, как оно устроено, какие внешние сервисы нужны и где искать основные части кода. Из-за этого перед каждой новой задачей приходится заново читать структуру проекта и восстанавливать архитектуру по классам.

Дополнительно заметны технические риски:

- `docker-compose.yml` содержит настройки `ollama` на верхнем уровне, вне `services`; это невалидная структура Compose-файла для конфигурации приложения.
- В части файлов русские комментарии и сообщения выглядят как mojibake (`РќР°...`), что усложняет чтение и поддержку.
- В репозитории есть рабочие данные в `storage/` и сборочные артефакты в `target/`; их стоит не смешивать с исходным кодом при анализе.

## 2. Как исправлять

Для дальнейшей работы нужен стабильный набор контекстных файлов:

- этот файл описывает назначение и архитектуру проекта;
- `docs/RUNBOOK.md` описывает запуск, зависимости и проверку;
- `docs/API_CONTEXT.md` фиксирует основные HTTP-точки;
- `docs/KNOWN_ISSUES.md` собирает найденные проблемы и улучшения.

Такой контекст позволит дальше менять код точечно: сначала сверяться с архитектурой и контрактами, затем исправлять конкретную проблему, после этого обновлять документацию.

## 3. Что представляет собой проект

RagAnalyzer - Spring Boot приложение для RAG-анализа документов. Пользователь загружает документы, приложение извлекает текст, режет его на чанки, строит эмбеддинги через Ollama, сохраняет чанки в PostgreSQL с расширением pgvector и отвечает на вопросы по выбранным документам.

Основной сценарий:

1. Пользователь загружает `.txt`, `.pdf` или `.docx`.
2. `DocumentService` сохраняет файл через `FileStorageService`.
3. `TextExtractionService` извлекает текст.
4. `ChunkService` делит текст на фрагменты.
5. `EmbeddingService` запрашивает эмбеддинги у Ollama.
6. `VectorStoreService` сохраняет вектор в `document_chunks.embedding`.
7. `SearchService` и `ChatService` ищут похожие чанки.
8. `LlmService` отправляет найденный контекст в Ollama и возвращает ответ.

## Ключевые директории

- `src/main/java/ru/max/raganalyzer` - backend-код приложения.
- `src/main/resources/db/migration` - Flyway-миграции базы данных.
- `src/main/resources/static` - статический фронтенд.
- `storage/documents` - локальное хранилище загруженных документов.
- `storage/images` - извлеченные изображения страниц/документов.
- `target` - Maven-сборка, не источник истины.

## Основные backend-слои

- `controller` - REST API.
- `service` - бизнес-логика: документы, чанки, эмбеддинги, поиск, чат, LLM.
- `repository` - Spring Data JPA репозитории.
- `entity` - JPA-сущности: документы, чанки, изображения.
- `dto` - контракты запросов и ответов.
- `config` - настройки `app.storage`, `ollama`, `rag.search`.
- `client` - DTO для общения с Ollama API.

## Внешние зависимости

- Java 21.
- Spring Boot 3.5.15.
- PostgreSQL 16 с pgvector.
- Flyway для миграций.
- Apache Tika для извлечения текста.
- Ollama для embedding и chat/generate.

## Текущие модели по умолчанию

Настройки находятся в `src/main/resources/application.yml`:

- embedding: `nomic-embed-text:latest`
- chat: `qwen2.5:7b`
- Ollama URL: `http://localhost:11434`
- RAG top-k: `8`
- max-distance: `0.9`

## Что важно помнить перед изменениями

- Размерность pgvector в первой миграции: `vector(768)`. Модель эмбеддингов должна возвращать векторы такой же размерности.
- `spring.jpa.hibernate.ddl-auto=validate`, поэтому структура БД должна соответствовать сущностям и миграциям.
- Чат требует список выбранных `documentIds`; без него запрос считается некорректным.
- Стриминг ответа реализован через `SseEmitter` и endpoint `/api/chat/ask/stream`.
